## 1. Sentry Crash Reporter — Account & DSN

**What:** Provision a Sentry project, obtain the DSN, configure the opt-in telemetry consent flow in the UI, and verify symbolicated backtraces arrive for a test panic.

**Why it's blocked:** `src-tauri/src/` has no Sentry integration. ROADMAP.md marks this `[DEFERRED — external]`. Code cannot be written until a DSN exists.

**Prerequisites:**
- Company credit card or budget approval (Sentry Team plan ~$26/mo for 100k errors)
- Decision on data residency (US vs EU region)

**Steps:**

1. **Create Sentry account**
   - Go to sentry.io → "Start for Free" or sign in with the company Google/GitHub account
   - Choose organisation name: `crossterm` (or your org)
   - Select region: EU if privacy policy targets EU users; US otherwise

2. **Create a Sentry project**
   - Platform: `Rust`
   - Project name: `crossterm-desktop`
   - Copy the DSN — it looks like `https://<key>@o<org>.ingest.sentry.io/<project>`

3. **Add DSN to Tauri secrets**
   - Add to `src-tauri/.env` (not committed): `SENTRY_DSN=<dsn>`
   - In `src-tauri/tauri.conf.json` or `build.rs`, read the env var at build time
   - Wire into the `sentry` crate initialisation in `src-tauri/src/main.rs`

4. **Implement opt-in consent flow** (this is a code step — but requires the DSN first)
   - Add `telemetry_enabled: bool` to `Settings`
   - Show a one-time "Help improve CrossTerm?" dialog on first launch (after the first-run wizard)
   - Only initialise Sentry if `telemetry_enabled == true`

5. **Verify symbolicated backtraces**
   - In a debug build, add a temporary `panic!("test sentry")` behind a hidden keyboard shortcut
   - Fire it, check sentry.io dashboard within 60 seconds for the event
   - Confirm the stack trace shows Rust function names (not hex addresses)
   - Remove the test panic before release

6. **Set up Sentry release tracking**
   - In `release.yml`, add a `sentry-cli releases finalize $VERSION` step after the build
   - Upload debug symbols: `sentry-cli upload-dif target/release/crossterm`

**Estimated effort:** 4–6 hours (account setup + code changes + verification)

## 2. Okta / Azure AD — OIDC Live Smoke Test

**What:** Run the OIDC PKCE loopback flow against a real Okta developer tenant and an Azure AD tenant to confirm the implementation works end-to-end.

**Why it's blocked:** OIDC PKCE is fully implemented. ROADMAP.md: `Okta + Azure AD tested with real accounts — deferred (requires external IdP access)`. No code change is needed — this is pure smoke-test validation.

**Prerequisites:**
- Okta developer account (free at developer.okta.com)
- Azure account (free tier at portal.azure.com) or access to an existing Azure AD tenant
- A test user in each IdP

**Steps — Okta:**

1. **Create Okta developer account**
   - Sign up at developer.okta.com (free, no credit card)
   - Domain will be `https://dev-<number>.okta.com`

2. **Create an OIDC application in Okta**
   - Applications → Create App Integration → OIDC → Native Application
   - Sign-in redirect URI: `http://127.0.0.1` (with any port — Tauri uses ephemeral port)
   - Sign-out redirect URI: `http://127.0.0.1`
   - Copy Client ID

3. **Configure CrossTerm**
   - In Settings → Enterprise → SSO, enter:
     - Provider: Okta
     - Issuer URL: `https://dev-<number>.okta.com/oauth2/default`
     - Client ID: (from step 2)

4. **Run smoke test**
   - Click "Sign in with SSO" in CrossTerm
   - A browser window should open to the Okta login page
   - Sign in with a test user
   - CrossTerm should complete the PKCE exchange and show the signed-in state
   - Check that `profile.sso_sub` is populated in the config file

5. **Document result** — pass/fail + any error in `docs/SSO_SMOKE_TEST.md`

**Steps — Azure AD:**

1. **Create an App Registration in Azure AD**
   - portal.azure.com → Azure Active Directory → App registrations → New registration
   - Redirect URI: `http://localhost` (type: Public client/native)
   - Under Authentication, enable "Allow public client flows"
   - Copy Application (client) ID and Directory (tenant) ID

2. **Configure CrossTerm**
   - Issuer URL: `https://login.microsoftonline.com/<tenant-id>/v2.0`
   - Client ID: (from step 1)

3. **Run smoke test** (same procedure as Okta)

**Estimated effort:** 2–3 hours total (account creation + config + two smoke tests)

## 3. External Penetration Test

**What:** Engage a security firm to conduct a black-box / grey-box penetration test of CrossTerm, focusing on: vault crypto, OIDC/SAML flows, plugin WASM sandbox, and the network scanning surface.

**Why it's blocked:** ENGINEERING_PLAN.md Phase 3 exit gate: `[DEFERRED — engagement] External penetration test. Requires contracting a security firm. Recommended before v1.0 GA.`

**Prerequisites:**
- Budget approval (typical external pen test: $8,000–$25,000 depending on scope and firm)
- A packaged, installable build of CrossTerm (pre-release build is fine)
- NDA signed with the firm

**Steps:**

1. **Define scope document** — write a 1-page brief covering:
   - In scope: vault encryption/decryption, OIDC login flow, plugin capability sandbox, network scanner, SFTP file operations
   - Out of scope: underlying OS, third-party SSH servers
   - Target platforms: macOS (primary), Windows
   - Deliverable: written report with CVSS-scored findings and remediation recommendations

2. **Solicit quotes from 3 firms** — recommended shortlist:
   - Cure53 (Berlin — strong application sec track record)
   - NCC Group (UK/US — enterprise preferred)
   - Doyensec (boutique, app sec focused)
   - Alternative: post on Bugcrowd/HackerOne as a private program if budget is tight

3. **Provide test build**
   - Package a signed (but not yet publicly released) build
   - Include a `PENTEST_BRIEF.md` with architecture overview, known limitations, and focus areas from `docs/THREAT_MODEL_VAULT.md`

4. **Conduct test** — typical duration: 5–10 business days

5. **Triage findings**
   - Critical/High: block v1.2 GA until patched and verified
   - Medium: fix before v1.2; can be in a patch release
   - Low/Informational: add to backlog; note in release notes

6. **Publish summary** (optional but builds trust) — a sanitized "Security Audit Summary" page on the CrossTerm website

**Estimated effort:** 2–4 weeks elapsed (procurement + test window + remediation)

## 4. SOC 2 Type I Audit Engagement

**What:** Engage a licensed CPA auditor to conduct a SOC 2 Type I readiness assessment and audit, producing a report that enterprise customers can request.

**Why it's blocked:** ENGINEERING_PLAN.md: `[DEFERRED — auditor] SOC 2 Type I audit. Requires engaging a licensed CPA auditor. Pre-audit readiness work (policy docs, access reviews) can begin in Phase 2.`

**Prerequisites:**
- Budget: SOC 2 Type I audits typically cost $15,000–$40,000
- At minimum 3 months of operational history with documented controls
- Completed access review and policy documentation

**Pre-audit readiness (can be done before engaging auditor):**

1. **Write required policy documents** (if not already done):
   - Information Security Policy
   - Change Management Policy
   - Incident Response Plan
   - Access Control Policy
   - Vendor Management Policy

2. **Evidence collection system** — use Drata, Vanta, or SecureFrame to automate evidence collection from GitHub, AWS, etc. (cost: ~$500–$1,500/mo)

3. **Access review** — document who has access to: production systems, GitHub repo, any cloud accounts; verify principle of least privilege

**Steps to engage auditor:**

1. **Select a SOC 2 audit firm**
   - Johanson Group (online-first, lower cost)
   - Schellman (enterprise)
   - A-LIGN (mid-market)
   - Leverage your compliance automation vendor's auditor network if using Drata/Vanta

2. **Scoping call** — define the Trust Service Criteria in scope (minimum: Security; optionally Availability, Confidentiality)

3. **Observation period** — Type I is a point-in-time assessment; no minimum period required (vs. Type II which requires 6+ months)

4. **Provide evidence package** — system descriptions, control narratives, screenshots of security controls in action

5. **Report issuance** — target: unqualified opinion (clean report)

**Estimated elapsed time:** 6–10 weeks from engagement to report issuance  
**Estimated total cost:** $20,000–$35,000 (audit + compliance tooling for 1 year)

## 5. Memory Profiling Baseline

**What:** Run heaptrack (Linux) or Instruments (macOS) against CrossTerm to establish a memory usage baseline, then identify the top-3 allocations in SSH scrollback and SFTP transfer queue.

**Why it's blocked:** ROADMAP.md: `[DEFERRED — baseline needed] Memory profiling pass. Requires: heaptrack/Instruments baseline on a reference device.`

**Prerequisites:**
- M2 MacBook Air (8 GB RAM) — the reference device from `ENGINEERING_PLAN.md §6.1`
- Xcode installed (for Instruments)
- A representative workload script (described below)

**Steps — macOS with Instruments:**

1. **Build a release build with debug symbols**
   ```
   cargo build --release
   # In src-tauri/Cargo.toml, ensure:
   [profile.release]
   debug = 1
   ```

2. **Open Instruments**
   ```
   open /Applications/Xcode.app/Contents/Applications/Instruments.app
   ```
   - Template: "Allocations"
   - Target: CrossTerm.app

3. **Run the workload script** — perform the following in sequence while Instruments is recording:
   a. Open CrossTerm cold
   b. Unlock vault (master password)
   c. Connect to 3 SSH sessions simultaneously
   d. Scroll each terminal back 10,000 lines (use `seq 1 10000` to generate output)
   e. Open SFTP panel on one session; transfer a 100 MB file both directions
   f. Disconnect all sessions
   g. Close CrossTerm

4. **Identify top allocations**
   - In Instruments → All Allocations, sort by "Persistent Bytes"
   - Record top 5 allocation sites with their stack traces
   - Focus on: `ssh/` terminal buffer allocations, `sftp/` transfer queue

5. **Record baseline numbers** in `docs/PERF_BASELINE.md`:
   - Peak RSS at steady state (3 sessions open, 10k scrollback)
   - Allocations from `TerminalBuffer` or equivalent
   - Allocations from SFTP transfer path

6. **Open GitHub issues** for the top-3 allocations identified, tagged `perf`

**Steps — Linux with heaptrack:**

```bash
sudo apt install heaptrack heaptrack-gui   # Ubuntu
heaptrack crossterm &
# ... run workload ...
heaptrack --analyze heaptrack.crossterm.<pid>.zst
heaptrack_gui heaptrack.crossterm.<pid>.zst
```

**Estimated effort:** 3–4 hours (build + profile run + analysis + issue filing)

## 6. Onboarding Timed Walkthrough

**What:** Have a real "new user" (someone who has never used CrossTerm) attempt to connect to 3 SSH hosts starting from a fresh install, timing the process. Target: < 5 minutes. ENGINEERING_PLAN.md Phase 1 exit gate.

**Why it's blocked:** `[DEFERRED — human QA] Onboarding test (< 5 min to first connected session). Requires a timed walkthrough with a new user on a clean machine.`

**Prerequisites:**
- A clean macOS machine (or VM) with no prior CrossTerm installation
- 2–3 SSH hosts with known credentials (can be local VMs)
- A willing participant who has not seen CrossTerm before
- A screen recorder (QuickTime or OBS) running

**Steps:**

1. **Prepare the test environment**
   - Install a fresh macOS (or use a VM snapshot)
   - Install CrossTerm from the `.dmg` — do NOT pre-configure anything
   - Set up 3 SSH targets (can be local `sshd` processes on different ports)
   - Write down credentials for the participant on a card

2. **Brief the participant** (script):
   > "You've just installed CrossTerm. Your goal is to connect to the 3 SSH servers listed on this card. I'll time you from when you first open the app. Ask me nothing — treat me as if I'm not here. There is no wrong answer."

3. **Start the timer and recording**
   - Note time T0 when the app first opens
   - Note time T1 when the first SSH session is connected
   - Note time T2 when all 3 sessions are connected

4. **Record observations:**
   - Where did the participant pause or look confused?
   - Did they use the import wizard?
   - Did they find the vault creation step intuitive?
   - Were any error messages shown?

5. **Debrief** (5 min post-test):
   - "What was the most confusing part?"
   - "What would you have expected to see but didn't?"
   - "On a scale of 1–10, how confident are you that CrossTerm is secure?"

6. **Document results** in `docs/ONBOARDING_TEST_RESULTS.md`:
   - Total time T0 → T2
   - Top 3 friction points with screenshots
   - Pass/fail against < 5 minute target

7. **Repeat with at least 2 more participants** (3 total minimum for signal)

**Estimated effort:** 2 hours per participant including prep (6 hours total for 3 participants)

## 7. Windows CI Runner — GitHub Actions

**What:** Add a self-hosted Windows GitHub Actions runner (or enable `windows-latest` hosted runner) so the PuTTY registry reader and Windows-specific paths are tested in CI.

**Why it's blocked:** ENGINEERING_PLAN.md: `PuTTY registry reader — CI test on Windows deferred until Windows runner added`. Hosted `windows-latest` runners cost ~$0.016/min (vs $0.008 for Linux).

**Prerequisites:**
- GitHub repository admin access
- Decision: hosted runner (`windows-latest`) vs self-hosted Windows machine

**Steps — Option A: Use GitHub-hosted `windows-latest` (recommended for simplicity):**

1. **Edit `.github/workflows/ci.yml`**
   - Add a new job `test-windows`:
     ```yaml
     test-windows:
       runs-on: windows-latest
       steps:
         - uses: actions/checkout@v4
         - uses: dtolnay/rust-toolchain@stable
         - run: cargo test --lib -- --test-output immediate
           working-directory: src-tauri
     ```
   - Estimated cost: ~15 min × $0.016 = $0.24 per CI run

2. **Add a PuTTY registry test fixture** (code step — but requires the runner first):
   - In `src-tauri/src/importer/mod.rs`, add `#[cfg(target_os = "windows")]` test that writes fake PuTTY registry keys and verifies `parse_putty_registry()` reads them

3. **Verify the run passes** in GitHub Actions → the `test-windows` job should show green

**Steps — Option B: Self-hosted runner (cost: $0, but requires hardware):**

1. A Windows machine (physical or VM) with:
   - Windows 10/11 (64-bit)
   - Git, Rust toolchain, Node.js 20 installed

2. Register the runner:
   - GitHub repo → Settings → Actions → Runners → New self-hosted runner
   - Follow the PowerShell instructions provided

3. Update `ci.yml` to use `runs-on: [self-hosted, windows]`

**Estimated effort:** 2–3 hours (Option A); 4–6 hours (Option B)

## 9. Apple Developer Account & macOS Notarization

**What:** Ensure an Apple Developer Program membership is active, that CrossTerm is notarized on every release tag, and (future) that the iOS project can be submitted to App Store.

**Why it's needed:** macOS Gatekeeper will block CrossTerm for all users if the app is not notarized. ENGINEERING_PLAN.md Risk Register: `macOS notarisation / Gatekeeper policy changes block distribution — Critical impact`.

**Prerequisites:**
- Apple ID (personal or company)
- $99/year Apple Developer Program fee
- macOS machine for notarization (can be the CI Mac runner)

**Steps:**

1. **Enroll in Apple Developer Program**
   - Go to developer.apple.com/programs/enroll/
   - Choose Individual or Organisation (Organisation requires DUNS number — allow 2 weeks)
   - Pay $99/year

2. **Create a Developer ID Application certificate**
   - Xcode → Settings → Accounts → Manage Certificates → Add Developer ID Application
   - Export as `.p12` with a strong password

3. **Store credentials as GitHub Actions secrets**
   - `APPLE_CERTIFICATE` — base64 of the `.p12` file: `base64 -i cert.p12`
   - `APPLE_CERTIFICATE_PASSWORD` — the `.p12` export password
   - `APPLE_ID` — your Apple ID email
   - `APPLE_APP_PASSWORD` — an app-specific password from appleid.apple.com
   - `APPLE_TEAM_ID` — your 10-character Team ID from developer.apple.com

4. **Configure Tauri notarization** in `src-tauri/tauri.conf.json`:
   ```json
   "macOS": {
     "signingIdentity": "Developer ID Application: Your Name (TEAMID)",
     "notarizationCredentials": {
       "appleId": "$APPLE_ID",
       "password": "$APPLE_APP_PASSWORD",
       "teamId": "$APPLE_TEAM_ID"
     }
   }
   ```
   Tauri handles the `codesign` + `notarytool submit` + stapling automatically.

5. **Test notarization on the next tag push**
   - Push a test tag: `git tag v1.1.99 && git push origin v1.1.99`
   - Watch `release.yml` — it should call `tauri build --target universal-apple-darwin`
   - After the release, download the `.dmg` and run:
     ```bash
     spctl --assess -v /Applications/CrossTerm.app
     # Expected: "accepted" source: notarized
     ```

6. **For iOS (future):**
   - Requires a separate Xcode project (out of scope for this Tauri repo)
   - Requires an App Store Connect app record
   - See ROADMAP.md Phase 5: `[DEFERRED — separate project] iOS app`

**Estimated effort:** 1–2 days for enrollment + CI wiring  
**Recurring:** $99/year

## 13. macOS Notarization on Every Tag Push

**What:** Verify that the `release.yml` workflow notarizes and staples every tag push automatically without manual intervention. This is a recurring release-time verification, not a one-time setup.

**Why it matters:** ENGINEERING_PLAN.md Risk Register: `macOS notarisation / Gatekeeper policy changes block distribution — Critical impact. Maintain Apple developer account in good standing; test notarisation on every tag.`

**This step depends on Step 9 being complete first.**

**Verification checklist (run after every release tag):**

```bash
# 1. Download the released .dmg from GitHub Releases
# 2. Mount it
hdiutil attach CrossTerm-v1.x.y-universal.dmg

# 3. Check Gatekeeper acceptance
spctl --assess -vv /Volumes/CrossTerm/CrossTerm.app
# Expected output:
# /Volumes/CrossTerm/CrossTerm.app: accepted
# source=Notarized Developer ID

# 4. Check stapling
xcrun stapler validate /Volumes/CrossTerm/CrossTerm.app
# Expected: The validate action worked!

# 5. Check signature
codesign --verify --deep --verbose=4 /Volumes/CrossTerm/CrossTerm.app
# Expected: valid on disk

# 6. Eject
hdiutil detach /Volumes/CrossTerm
```

**If notarization fails:**
- Check `release.yml` logs for `notarytool` error
